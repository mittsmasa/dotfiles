# slide-outline

素材テキストからスライドの流れ(骨子)を `slide.md` に作る。または既存の骨子をレビューする。

**この skill がやらないこと**: html/pdf 化。見た目のデザイン。それらは骨子が固まった後の別フェーズ(`slide` skill 等)に送る。骨子段階で紙面の話を始めると、流れの議論が止まるため。

## モード判定

- 素材(テキスト、メモ、口述)があって骨子がまだない → **生成モード**
- すでに slide.md 相当の骨子がある → **レビューモード**
- 両方ある(素材と骨子)なら、レビューモードで素材との突合まで行う

## 生成モード

### Step 1: タイトル・ターゲット・コアメッセージを最初に固める

書き始める前に、この3つをユーザーと確定し、slide.md の冒頭にメタ情報として明記する。

- **タイトル**: 必須。決まっていなければ「後で決める(仮)」と置く。空欄にしない(タイトルスライドが構成から漏れるため)。スライドの1枚目は必ずタイトルスライド
- **ターゲット**: 誰に向けて話すか。「社内の企画・デザイン・開発・QAのメンバー」のように具体的に
- **コアメッセージ**: 聞き手に持ち帰ってほしい一文。提案なら「〜しませんか」まで含める

なぜ最初か: 以降のすべての取捨選択(このスライドは要るか、この表現は届くか)の判断基準になるため。素材から推定できるなら案を出して確認する。推定できないなら聞く。

### Step 2: 見出しの列だけでアウトラインを作る

本文を書く前に、スライドの見出しだけを並べて流れを確認する。**見出しだけ読んで話が通るか**が合格基準。通らないなら本文を書いても直らない。

構成はコアメッセージから逆算する。「この一文を聞き手に持ち帰らせるには、どの順で何を見せる必要があるか」を考えると、必要な章は自ずと絞られる。型は出発点であって鋳型ではない。素材を型に押し込まない。

よく使う型の例:

- **課題解決・提案系**(「〜しませんか」で終わる話):
  共感オープニング(問いかけ) → Before: いまの苦しみ(具体) → 構造分析(誰も悪くない、構造の問題) → 転換点 → After: 提案するあり方(Before と1対1対応) → 難所 → 最初の一歩(start small) → まとめ
- **学び共有・体験談系**(「やってみたらこうだった」):
  何をやったか → 何が起きたか(事実、失敗含む) → そこから引いた学び(抽象化) → 聞き手が明日使える形に翻訳 → まとめ
- **技術・概念の解説系**(「これを知ってほしい」):
  なぜ今この話か(聞き手との接点) → 全体像を一枚で → 具体(コード、デモ、例) → 使いどころと限界 → まとめ

型が違っても効く背骨は共通している:

- 聞き手の関心・体験から開く(共感か問い)。自分の話したいことから開かない
- 具体と抽象を往復する。具体だけでは持ち帰れず、抽象だけでは刺さらない
- 限界・難所・失敗を正直に語る。隠した瞬間、聞き手は提案全体を疑い始める
- 聞き手が動ける小さな一歩で閉じる

### Step 3: 各スライドを「載せる」と「ノート」に分けて埋める

```markdown
## スライドの見出し

載せる(箇条書き + 表):

- キーワードだけの短い箇条書き

(図: 図やレイアウトの説明。html化フェーズへの申し送り)

**Key Message(最下部): このスライドで言いたい一文**

ノート: 口頭で話す内容。素材の詳しい文はこちらへ (→ 素材の対応箇所への参照)
```

「載せる」には**表現形式を明記する**(箇条書き / 表 / 対比の箇条書き / 吹き出し / 図のみ / 目次 など)。html化の段階で表現を発明させると紙面ごとに見せ方がブレるし、ユーザーが骨子の段階で紙面の密度を判断できない。

- 1見出し = 1スライド。スライド間は `---` で区切る
- **紙面に載せるのはキーワードだけ。話す内容はノートに分離する**。全文を紙面に載せると文字が小さくなり、聞く発表ではなく読む資料になる。ただし資料単体でも大枠が掴める程度には載せる(発表を見ていない人も後から読む)
- Key Message は全スライドに必須ではない。台詞だけで立つスライド(オープニング等)には置かない。置くときは**紙面の最下部で統一**する(位置がページごとに上下すると、聞き手が毎回視線の置き場を探すことになる)
- **本文と Key Message で同じことを言わない**。同じ主張が箇条書きと Key Message の両方にあるなら、本文側を削って Key Message に言わせる。重複は紙面を薄めるだけでなく、聞き手に「どちらが結論か」を迷わせる
- **列挙は図と本文で二重にしない**。情報源・ツール名などの列挙が図に入るなら、本文の箇条書きでは繰り返さず図に任せる。本文は「その列挙が何を意味するか」だけ言う
- 情報量が薄すぎるスライドは、骨子の段階で隣のスライドと統合する。html化のレイアウトで埋めようとしても余白が間延びするだけ
- 逆に、1枚割くほどではないがゼロにもしたくない論点(難所の詳細等)は、スライド化せず関連スライドのノートに「質疑・口頭で扱う」と退避する選択肢がある
- 素材に対応箇所があれば `(→ essence.md 6章)` のように参照を残す。骨子とソースの突合を後からできるようにするため

### Step 4: 中扉(セクション区切り)を入れる

章の切り替わりに目次スライドを置き、現在地を示す。

```markdown
## 中扉2: 転換点

1. Before: いまの苦しみ ✓
2. **転換点: AIエージェントの登場**
3. After: 企画のつくりかたをこう変えたい
4. 提案: 難所と最初の一歩
```

- **全章を最初から見せる**。伏せ字(???)で未来の章を隠す演出はしない。もったいぶっても得るものは少なく、聞き手はむしろ全体の地図を早く欲しがる
- 現在地は太字、話し終えた章は ✓。「← イマココ」のような注記は重ねない

### Step 5: 図を提案する

文字だけで伝わりにくいスライドには `(スライド案: ...)` で図を添える。効くパターン:

- **同じ図の再掲・積み増し**: オープニングのプロセス図に、Before で吹き出しが増えていく。壁の図が After で「壁を越える」図になる。図の再登場が「あの話がこう変わる」の対比を作る
- **距離・ギャップの可視化**: 問題の2点(発覚地点と修正地点など)が離れていることを軸上に置いて見せる
- **分岐のあるループはフローチャートで**: 「レビューで NG なら改善して戻る / OK なら先へ」のような判定を含むループは、循環図ではなくフローチャート(処理 → 判定 → NG/OK 分岐)で描く。循環図は「必ず回り続ける」ように読めてしまい、出口があることが伝わらない
- **要素の並び順を、話の登場順に揃える**: 図の中の列挙(情報源など)は、対応する概念がこれまで語られてきた順(役割順など)に並べる。文言の統一だけでなく順序も対応関係の一部
- **知識・データの「ありか」をアイコンで添える**: どのツールにあるか(Confluence / Figma / GitHub 等)を図に添えると実在感が出る。描き方(ロゴ画像は使わない等)は html 化フェーズの領分なので、骨子では「ありかをアイコンで」とだけ申し送る

図はまとめて提案し、ユーザーに選んでもらってから反映する。図の趣味は分かれるため。

### 協働のスタイル

ユーザーが対話的に進めたい様子(口述で1枚ずつ足してくる等)なら、一気に完成させず増築に付き合う。ワンショットの依頼なら全体を出してからレビューをもらう。**ユーザーが口述した台詞・言い回しは生々しさが価値**なので、整えすぎない(下記トーン参照)。

## トーンと言葉の守りごと

- **台詞スライドは生のまま**: 「この案、実現しようとすると既存のデータ構造が歪みそう...」のような現場の台詞は、文法的に整えると共感が死ぬ。そのまま残す
- **地の文には concise-japanese-writing を適用**: 一文一義、結論先出し、冗長表現の削除。台詞は適用対象外
- **他者への批判に読めないか**: 特定の役割・チームの苦しみを描くとき、「あの人たちができていない」と読める表現になっていないか。構造の問題として書く。発表は聞き手全員の前で読まれる
- **用語のダブルミーニング回避**: 同じ語が2つの意味を持ったら呼び分ける(例: 役割名の「開発」と全体の「プロダクト開発」)。聞き手は文脈で読み分けてくれない
- **未実践のことを実践済みのように書かない**: 構想段階なら「〜になる(はず)」「まだ実践していない構想です」と明示する。誠実さは提案の信頼の土台で、盛った瞬間に難所の議論もできなくなる
- **擬人的・情緒的な演出語を使わない**: 「AIが苦しみを癒す」のような言い回しは、即物的な動詞(「知識の収集と企画案の生成をサポートする」)に置き換える。演出は共感より照れを生み、技術的な提案の信頼を下げる
- **AI文体を消す**: 「重要なのは」「不可欠」「掘り下げる」等の空句、過剰な太字、対句の決め台詞連発を避ける

## レビューモード

既存の骨子に対して、以下を順に確認して指摘する。指摘は「何が問題か + どう直すと良いか」を対で。

1. **タイトル・ターゲット・コアメッセージが冒頭に明記されているか**。ないなら最初にそれを聞く(以降の指摘の基準になるため)。タイトルスライドが1枚目にあるか
2. **見出しだけ読んで流れが通るか**。見出しを抜き出して並べ、話が繋がるか確認する
3. **各スライドがコアメッセージに奉仕しているか**。どのメッセージにも効いていないスライドは削除候補
4. **素材との突合**(素材がある場合): 素材にあって骨子にない要素を列挙し、入れるべきか意図的な省略かをユーザーに確認する
5. **表現形式が決まっているか**: 各スライドの「載せる」に形式(箇条書き / 表 / 図 / 目次 等)が明記されているか
6. **トーンと言葉の守りごと**(上記セクション)に反していないか
7. **1スライド1メッセージか**: 2つ以上のことを言っているスライドは分割候補。逆に情報量が薄すぎるスライドは統合候補
8. **紙面とノートが分離されているか**: 話す内容がそのまま紙面に載っていたら、キーワード化してノートへ逃がす

紙面の注意(下部に重要文字を置かない、フォントサイズ、コントラスト)は骨子レビューでは扱わず、html化フェーズへの申し送りとしてメモに留める。

## 完成の目安

- 見出しだけ読んで話が通る
- 全スライドがコアメッセージに効いている
- 素材の主要素が入っているか、省いた理由を説明できる
- ユーザーが「この流れで話せる」と言える
