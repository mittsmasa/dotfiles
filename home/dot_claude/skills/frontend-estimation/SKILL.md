# Frontend Estimation

企画書・デザインからフロントエンドタスクの見積もりCSVを作成する。

## Workflow

### Step 1: 入力の収集

以下を収集する。不足があればユーザーに確認する。

- **参照CSVファイル**: 既存の見積もりCSVを読み、フォーマット（列名・記入規則・粒度感）を把握
- **企画書**: Confluence（Atlassian MCP）、Google Docs、または直接テキスト
- **デザイン**: Figma MCP があれば参照。利用不可ならユーザーに伝える
- **CSV列の記入ルール**: 各列に何を入れるかユーザーに確認（バージョン名、ラベル体系、コンポーネント名など）。ストーリーポイントはユーザーが記入する場合もある

### Step 2: 企画書の読み取り

Atlassian MCPが利用可能な場合:
1. `getAccessibleAtlassianResources` でcloudIdを取得
2. `getConfluencePage` でページ本文を取得（contentFormat: "markdown"）
3. 子ページがあれば `getConfluencePageDescendants` で確認

企画書から抽出すべき情報:
- 機能要件（画面単位で整理）
- 新規ステータス・データ定義
- UIの仕様（バリデーション、制約、表示条件）
- スコープ外の明示
- Figmaリンクがあれば記録

### Step 3: コードベース調査

Explore agentまたは直接調査で以下を把握する:
- フレームワーク・主要ライブラリ
- 既存の関連画面・コンポーネント
- 再利用可能なUIコンポーネント（テーブル、モーダル、フォーム部品等）
- API呼び出しパターン（GraphQL / REST）
- 既存に**ない**が新規に必要なもの（→ 工数増の根拠）

### Step 4: タスク分解

参照CSVのタスク粒度を参考に、画面単位でタスクを分解する。

分解の原則:
- **画面名は区切り行**として挿入（他の列は空）
- **1タスク = 1つの独立した成果物**（コンポーネント、API連携、バリデーション等）
- 「見た目only」と「振る舞い」は分けることが多い
- フェッチ（API連携）は独立タスクにする
- ページレイアウト（コンポーネントの組み合わせ）は独立タスクにする

共通タスクとして最後に:
- スキーマ・型定義の更新
- テスト
- QA・レビュー対応バッファ

### Step 5: CSV出力

参照CSVのフォーマットに厳密に合わせて出力する。`.plan/` ディレクトリに保存し、VSCodeで開く。

### Step 6: ユーザーレビュー

出力後、以下を伝える:
- タスク総数と画面ごとの内訳
- 不確定要素・リスク（新規コンポーネント開発、BE依存など）
- Open Questions

## SP基準の参考値

| SP | 目安 |
|----|------|
| 1 | 見た目のみの単純なコンポーネント |
| 2-3 | 標準的なコンポーネント + API連携 or フォーム動作 |
| 5 | フィルター動作やフェッチ処理など中程度の複雑さ |
| 8 | 新規コンポーネント設計 + 実装 |
| 13 | 高複雑度（テーブルのステート制御等） |

この基準はプロジェクトによって異なるため、参照CSVから毎回推定すること。
