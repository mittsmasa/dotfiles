# concise-japanese-writing

第三者が読むドキュメントとして残る日本語を Claude が書くときの作法。会話のトーンとは分離し、書き残されるテキストに対してのみ適用する。

## When to apply

**適用する**:
- PR description / PR の概要文
- PR コメント / レビューコメント（日本語）
- 提案書・企画書・設計書・要件定義書などのドキュメントのたたき
- Issue 本文、技術メモ、ADR など、ファイル/PR/チケットに残る日本語まとまり文

**適用しない**:
- 通常のチャット応答そのもの（出力スタイルが別途乗っている）
- コード内コメント（最小限主義は別ルール）
- ユーザーへのワンライン報告・進捗共有

## 核となる原則

各ルールには「なぜそうするか」を併記する。理由がわかっていれば、エッジケースで機械的に当てはめず、判断できる。

1. **一文一義**: 一文にひとつのことだけ書く。複数の主張を詰めない
   - なぜ: 日本語は述語が文末に来るため、一文に複数の主張があると、読み手は文末まで意味を確定できず認知負荷が上がる。一義に絞れば、文の途中で意味が決まる。
2. **一文を短く**:
   - 一般文書（提案書・設計書）: 40〜60 字を目安
   - PR description / PR コメント / Slack 的な文脈: 20〜40 字を目安
   - 「やさしい日本語」（24 字以下）までは縮めない。技術文書では硬すぎる
   - なぜ: 長くなるほど主述のねじれや係り受けの誤読が増える。新聞も 40〜60 字を上限の目安にしている。短さは正確さを下げない、むしろ上げる。
3. **結論先出し**: 段落の頭にトピックセンテンス（その段落の主張）を置く。理由・詳細・背景はあと
   - なぜ: 読み手は先に結論を知ることで、続く詳細を「結論を支える材料」として処理できる。順序が逆だと最後まで読まないと話題がわからない。
4. **主語と述語を近づける**。必要な主語は省かない
   - なぜ: 距離が離れるほど対応関係が見えにくくなり、主述のねじれや誤読が増える。省略された主語が前文と違う場合、誤読は致命的になる。
5. **修飾語は被修飾語の直前に置く**
   - なぜ: 離れると係り受けが曖昧になり、別の語にかかって読まれる。誤読は文末まで気付かれない。
6. **具体・数値で書く**: 「かなり」「適切に」「しっかり」より「30%」「3 件」「24h 以内」
   - なぜ: 抽象語は読み手によって解釈幅が広く、合意したつもりで実は認識がずれる事故を生む。数値・期日・固有名詞は解釈幅を狭める。

## アンチパターン（出力前に当てる）

書いた文に対して、以下を削除または書き換える。各項目の「なぜ」は、機械的な置換ではなくニュアンスを残す判断の根拠になる。

### 冗長表現
- 「〜ということ」「〜することができる」「〜という点において」「〜のような形で」
- 例: 「実装することができる」→「実装できる」/「〜という点において重要」→「〜が重要」
- なぜ: 情報量を増やしていないノイズ。読み手の時間を奪うだけで、削っても意味は変わらない。

### 動詞の名詞化（サ変動詞の過剰）
- 「実施する」「分離する」「確認する」が空疎な場面では和語に戻す
- 例: 「分離する」→「分ける」/「確認を行う」→「確認する」
- なぜ: 漢語サ変は形式ばって距離を作り、和語より字数も増える。具体性も失われやすい。

### 強調語の空打ち
- 「非常に」「とても」「かなり」「しっかり」「きちんと」が情報を増やしていないなら削る
- なぜ: 強調語を連発すると、本当に強調したい箇所が他に埋もれて弱まる。具体数値に置き換えるか、削る。

### 意味の重複
- 「あらかじめ予測する」「まず最初に」「過半数を超える」「各々それぞれ」
- なぜ: 同じ意味を 2 回読まされる。読み手は「何か違いがあるのか」と探して時間を使う。

### 二重否定
- 「〜しないこともない」「〜なくはない」→ 肯定で書き直す
- なぜ: 解釈が読み手に委ねられ、意図が曖昧になる。肯定形なら判断の幅が狭まる。

### 指示語の連発
- 「これ」「それ」が遠い／指示先が曖昧 → 名詞に戻す
- なぜ: 指示先が離れると、読み手は参照解決のために前を読み返す。誤った先を指していても気付かれにくい。

### 翻訳調の受動態
- 「〜が行われる」「〜がなされる」「〜することが求められる」→ 主体を立てて能動で書く
- 例: 「キャッシュの導入が行われる」→「キャッシュを入れる」
- なぜ: 主体が消えると、誰がやるのか、誰の責任かが曖昧になる。技術文書では行為者の特定が肝心。

### 抽象表現の放置
- 「適切に対応」「しっかり検討」「最適化を図る」「品質を担保する」→ 何をどうするか具体的に
- 例: 「パフォーマンスを最適化する」→「N+1 クエリを潰し、応答を 500ms 以下に抑える」
- なぜ: 抽象語は何も言っていないのと同じ。読み手は次のアクションを決められず、合意形成も検証もできない。

### 修飾語の盛りすぎ
- 形容詞・副詞が連鎖したら核に絞る
- なぜ: 装飾が増えるほど核の主張が埋もれる。読み手は「結局何が言いたいのか」を取り出す作業を強いられる。

### 3 行を超える一文
- 句点で分割する
- なぜ: 一文が長くなるほど主述のねじれが発生しやすく、係り受けも追いにくくなる。

### 主述のねじれ
- 修飾を全部取って主語＋述語だけにし、意味が通るか確認
- なぜ: ねじれた文は読み手に「何かおかしい」という違和感だけを残し、意味の再構築に余計な労力を要求する。

### 前置きの儀礼
- 「本書では〜について述べる」「以下に〜を記載する」「まず初めに〜から説明する」→ 削って本題から入る
- なぜ: 読み始めは集中力が最も高い。そこを儀礼で埋めるのは情報密度が低く、読み手の時間とアテンションの無駄。

## 文脈別のトーン

### PR description
- タイトルは要点 1 行（70 字以下）
- 本文の基本構成: `## Summary`（1〜3 箇条書き）+ `## Test plan`（チェックリスト）
- 背景・動機は必要なら 1〜2 行で。設計議論は別ドキュメント or PR コメントに分離
- リポジトリ既存テンプレートがあれば従う

### PR コメント / レビューコメント
- 1〜3 文。長くしない
- 「何が問題か」「どう直すと良いか」を端的に
- 曖昧さを残さない: 「気になる」→「ここで X が null のとき落ちる」
- 提案なら命令ではなく相談形: 「〜してはどうでしょう」「〜も検討候補です」

### 提案書・企画書
- 構成: 結論 → 根拠 → 詳細 → 補足。各章の頭にトピックセンテンス
- 「なぜ今やるか」を冒頭に置く
- 数値・期日・担当を具体化する

### 設計書 / ADR
- 構成: 背景・目的 → 決定事項 → 実装方針 → 代替案と棄却理由
- 「なぜ（背景）」「何を（決定）」「どう（実装）」「他に何があったか（代替案）」の順
- 代替案を棄却した理由は短くてよいが省略しない

### 要件定義書
- 主語を明示する。「ユーザーが」「システムが」を省略しない
- 「〜できる」（capability）と「〜しなければならない」（requirement）を混ぜない

## 自己チェック（出力前に内部で当てる）

1. 一文 60 字超えがないか（PR コメントなら 40 字超え）
2. 段落の頭に結論があるか
3. 冗長表現リストの語が残っていないか（「ということ」「することができる」など）
4. 「適切」「しっかり」「最適」「担保」など空虚な抽象語が無条件に置かれていないか
5. 主述のねじれがないか（主語＋述語だけにして読む）
6. 数値・固有名詞で具体化できる箇所が抽象のままになっていないか
7. 前置き・儀礼的な書き出しを削れないか
8. 受動態が連続していないか

## やらないこと（過剰適用の禁止）

- **用語の機械的置換でニュアンスを破壊しない**。ドメイン用語や慣用表現として定着しているものは残す
- **「やさしい日本語」レベル（24 字以下）まで縮めない**。これは行政文書向け基準で、技術文書では硬すぎる
- **ですます調 / である調はドキュメントの既存トーンに合わせる**。skill が一方的にどちらかへ寄せない
- **コード・コマンド・固有名詞・引用はそのまま**
- **議論の余地を残すべき箇所まで断定形にしない**。提案・相談はその形を保つ

## Before / After

### 例 1: PR description
Before:
> 本PRでは、ユーザー認証に関する処理において、これまで同期的に行われていた検証処理を非同期化することで、レスポンスの改善を図るための変更を実施しました。

After:
> ## Summary
> - ユーザー認証の検証を非同期化
> - p95 レスポンスを 800ms → 200ms に短縮（ステージング計測）

### 例 2: 設計書の一節
Before:
> キャッシュ層を導入することによって、データベースへのアクセス回数を削減することができるため、全体的なパフォーマンスの向上が期待されると考えられます。

After:
> Redis をキャッシュ層に挟み、DB アクセスを減らす。想定効果は応答時間 X ms → Y ms。

### 例 3: PR コメント
Before:
> このあたりの処理について、もう少し考慮した方がよいのではないかと思われる箇所が散見されるように感じます。

After:
> ここ、`user` が null のときに落ちそうです。早期 return を入れてはどうでしょう。

### 例 4: 提案書
Before:
> 弊チームにおきましては、現状のリリースプロセスについて様々な課題が認識されているところであり、これらを解決するために何らかの改善施策を実施することが必要であると考えております。

After:
> リリースプロセスを見直したい。理由は 3 点：
> 1. 手動デプロイで月 2 回ミスが出ている
> 2. リリース所要時間が平均 3h
> 3. ロールバック手順が属人化している
