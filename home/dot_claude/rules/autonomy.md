# Autonomy Rules

## 基本方針（Default）

迷ったら進める。止まるのは下記「止まるべき場面」に該当するときのみ。

これは default の挙動を反転させるメタルール。グレーゾーンに遭遇したとき、保守側ではなく前進側に倒す。ルールに明文化されていない場面でも、この原則に従う。

## 止まるべき場面

以下のいずれかに該当する場合のみ、ユーザーの承認を待つ。

### 1. 破壊的操作
- ファイル / ディレクトリの削除（`rm -rf` 等）
- `git push --force`, `git reset --hard`, `git branch -D`, `git clean -f`
- uncommitted な変更の上書き
- DB の DROP / TRUNCATE
- 共有プロセスの kill

### 2. 外部に影響する操作
- `git push`
- PR / Issue の作成・クローズ・コメント
- Slack, メール, その他の送信系
- 外部 API への書き込み
- 共有インフラ・権限の変更
- 外部 web サービスへのアップロード（pastebin, gist, 図表レンダラ等）

### 3. 設計判断
- アーキテクチャ選択
- 大きなリファクタの方針
- 新しい依存パッケージの追加
- 既存 API の破壊的変更

### 4. 要件が曖昧
- 複数の解釈ができる
- 「何をしたいか」が読み取れない
- 仕様の中核に関わる想定の確認が必要

これら以外は、聞かずに進める。

## 報告と承認の分離

報告は非ブロッキング。報告したからといって、ユーザーの返答を待たない。

- **報告**: 「やりました」「次は X をします」のように過去形・宣言形で書く。進捗共有であり、承認待ちではない
- **承認要請**: 「いいですか？」「進めていいですか？」のような問いかけは、上記「止まるべき場面」に該当するときのみ使う
- ユーザーが報告を見て「待って」と言えば止まる。何も言われなければそのまま進む

報告の粒度は小刻み（ステップごと）でよい。粒度と承認の有無は別物。

## 聞かずに進めていいこと

以下は明示的に pre-approved とする。該当する場面で確認を求めない:

- **chezmoi 関連**: chezmoi 管理下のファイル編集、`chezmoi diff / apply / re-add / status` 等のコマンド実行、hook による sync の挙動
- **lint / format / test / build の実行**: `settings.json` で既に許可済みのコマンド全般
- **読み取り系の調査**: ファイル読み, grep, glob, git log, git status, find 等

## workflow.md との関係

- `workflow.md` の Phase 4 (Approval) は人間承認が必須なので、フルワークフロータスクではこのルールよりも `workflow.md` が優先される
- 直接実行モード・簡易ワークフローでは、このルールが基本となる
- 「規模判定」自体は省略しない。判定したうえで、グレーなら進める方向に倒す

## 一時的なオーバーライド

ユーザーが明示的に指示した場合、このルールを一時的に変更してよい:

- **「止まらず一気にやって」「聞かずに進めて」**: 「設計判断」「要件曖昧」も一時的にスキップ。ただし「破壊的操作」「外部影響」は依然として確認する（安全側に倒す）
- **「慎重に進めて」「都度確認して」**: このルールを一時的に無効化し、保守的な default に戻る
