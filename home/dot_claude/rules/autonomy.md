# Autonomy Rules

## 基本方針

**迷ったら進める。** 止まるのは下記「止まるべき場面」に該当するときのみ。グレーゾーンでは保守側ではなく前進側に倒す。

## 止まるべき場面

以下のいずれかに該当する場合のみ、ユーザーの承認を待つ。

### 1. 破壊的操作
- ファイル / ディレクトリの削除（`rm -rf` 等）
- `git push --force`, `git reset --hard`, `git branch -D`, `git clean -f`
- uncommitted な変更の上書き
- DB の DROP / TRUNCATE
- 共有プロセスの kill

### 2. 外部に影響する操作
- `git push`
- PR / Issue の作成・クローズ・コメント
- Slack, メール, その他の送信系
- 外部 API への書き込み
- 共有インフラ・権限の変更
- 外部 web サービスへのアップロード（pastebin, gist, 図表レンダラ等）

### 3. 設計判断
- アーキテクチャ選択
- 大きなリファクタの方針
- 新しい依存パッケージの追加
- 既存 API の破壊的変更

### 4. 要件が曖昧
- 複数の解釈ができる
- 「何をしたいか」が読み取れない
- 仕様の中核に関わる想定の確認が必要

これら以外は、聞かずに進める。

## 報告と承認の分離

報告は非ブロッキング。報告したからといってユーザーの返答を待たない。

- **報告**: 「やりました」「次は X をします」のように過去形・宣言形で書く
- **承認要請**: 「いいですか？」は上記「止まるべき場面」に該当するときのみ
- ユーザーが「待って」と言えば止まる。何も言われなければそのまま進む

## 聞かずに進めていいこと（pre-approved）

- **chezmoi 関連**: chezmoi 管理下のファイル編集、`chezmoi diff / apply / re-add / status` 等の実行、hook による sync
- **lint / format / test / build の実行**: settings.json で許可済みのもの
- **読み取り系の調査**: ファイル読み, grep, glob, git log, git status, find 等

## workflow.md との関係

- フルワークフロータスクでは `workflow.md` の Phase 4 (Approval) が優先（人間承認必須）
- 直接実行モード・簡易フローでは本ルールが基本
- 「規模判定」は省略しない。判定したうえでグレーなら進める方向に倒す

## 一時的なオーバーライド

- **「止まらず一気にやって」「聞かずに進めて」**: 「設計判断」「要件曖昧」を一時スキップ。「破壊的操作」「外部影響」は依然として確認
- **「慎重に進めて」「都度確認して」**: default に戻る
